# Conclusiones del ejercicio 1: Compra SauceDemo E2E Serenity Screenplay
- Durante el desarrollo de la automatización se definieron dos features independientes: uno para el proceso de login y otro para el flujo de compra de productos. El proceso de compra depende obligatoriamente de una autenticación previa, el login fue reutilizado como paso inicial dentro del feature de compra, evitando así la duplicación innecesaria de escenarios. Por lo tanto, para una ejecución limpia y coherente, se asignaron tags diferenciados a cada feature y el runner fue configurado para ejecutar únicamente el flujo de compra, el cual incluye el login como prerrequisito. De esta manera el login se ejecuta una sola vez, reduciendo tiempo de ejecución y alineándose a las buenas prácticas en la automatización.
- El feature de login se mantiene para validaciones aisladas o futuras pruebas específicas, pero no se ejecutó en el flujo principal de compra para evitar redundancias.
- Todas las ejecuciones de esta automatización se han ejecutado correctamente de extremo a extremo, validando el flujo completo de login, selección de productos, listado en carrito, formulario de compra y confirmación de compra de dos productos.
- El uso de Screenplay está correctamente aplicado, lo cual permitió evidenciar una clara separación entre: Acciones del usuario (Tasks), Validaciones del sistema (Questions), Orquestación de escenarios (Steps).
- El reporte de Serenity generado es muy claro, proporciona una trazabilidad completa, permitiendo revisar de forma detallada y paso a paso cada acción de los escenarios. Estos reportes incluyen evidencia clara de la ejecución exitosa y capturas visuales de los pasos ejecutados, lo que facilita el análisis tanto funcional como técnico.
